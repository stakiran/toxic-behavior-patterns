# Toxic Behavior Patterns
[日本語](index.md) | [English](index_EN.md) | [GitHub](https://github.com/stakiran/toxic-behavior-patterns)

Toxic Behavior Patterns とは、チームにおいて自身の精神を脅かす「有害な振る舞い」または「人物のタイプ」を整理したものである。あらゆるチームのあらゆるメンバーに当てはまるほどの汎用性はないが、高確率で当てはまる程度には的を得ているはずだ。

いくつか注意点を述べておく:

- 本ドキュメントが想定する文脈は、0 to 1 ではなく 1 to 10 以上のチームワークである。つまり一個人による貢献よりもチームによる貢献でないとスケールしない状況にある、と考えてほしい
- 厳密な定義にまでは踏み込んでいないため、チーム内で認識を揃えるべきだ。たとえばブリリアント・ジャークの項では「イヤなヤツ」と書いているが、イヤなヤツの定義までは書いていない。メンバーの誰が見ても即座に納得できるようなシンプルな定義をつくってほしい。でないと、Toxic Behavior Patterns がイジメの免罪符や多様性の阻害要因になってしまう。

## ブリリアント・ジャーク（言いすぎ）
優秀だがイヤなヤツを指す。Netflix 発の概念であり、ソフトウェア開発の界隈では有名。

Netflix の定義では「チームワークを壊す振る舞い」とある。たとえば威圧的な言動、公開処刑的なフィードバック（皆がいる場で直接的にネガティブ・フィードバックをする）、誹謗中傷などハラスメントレベルの口調や言い回しなどが挙げられる。いずれにせよ、言動が「攻撃」と呼べるほどに鋭い。

対処としては、基本的に追放するべきだ。なぜなら、ブリリアント・ジャークは優秀なだけにプライドが高く、自らを改める視座に立つのがほぼ不可能だからである。ということは、建設的な対話ができないことを意味する。退職者が複数人発生するほどのインパクトがあるため、クリティカルな問題として追放に取り組むべきだ。

もし追放できないのなら、まずは隔離を目指す。ブリリアント・ジャークを n 人とコミュニケーションさせるのをやめ、代表者一人として行わせるようにする。つまり代表者が傘となってチームを守る。また意思決定権限をなくして助言者のポジションに収めたり、対面ではなくテキストで非同期コミュニケーションにしたりといった策も使える。とにかく、直接コミュニケーションする形を少しでも緩和していくことで、隔離に導いていく。

## エイリアン・センス（意味不明すぎ）
生理的な恐怖を感じるほど価値基準や知識水準があまりにもかけ離れていることを指す。

エイリアン・センスを持つ者は、チーム内で誰一人知らない概念を平然と持ち込んでくる。哲学や経済学といった学問寄りの言葉を持ち出してきたり、インフルエンサーの言葉の受け売りを強く主張してきたり、いきなり独自の定義や造語を出してきたりする。実は、この時点でチームプレイとしては破綻している。受け手のメンバーが知らないまま話を進めても理解できないし、かといって一から解説するとなると、それはそれで長大になってしまうからだ。チームプレイの原則の一つは共通言語――すでに定着している語彙と言語を使うことであり、要するにエイリアンは共通言語から外れた言語を持ち出してくるのである。

対処としては、エイリアンに共通言語を使うよう促すことだ。それで改善されるならば良しとする。逆に改善されないのであれば、ブリリアント・ジャークに格上げして、隔離や排除に向かった方が良い。ただし、ブリリアント・ジャークほど露骨に攻撃的ではないため、役割とコミュニケーションの仕方を配慮すれば、戦力として使えることも多い。おそらく通常のメンバーとは異なる「特殊な扱い」は必要だろう。この発想を持ち、実行できるだけの多様性があれば良い。ない場合は、エイリアンはほぼ「おとなしいブリリアント・ジャーク」と同義になる。

## エクスポーザー（晒しすぎ）
自己開示含む情報発信が多すぎる者を指す。

インターネット、オープンソース、インフルエンサー活動などに深く浸った者に多く、仕事におけるあらゆる情報と公開したがり、また自分に関するあらゆる情報を開示したがる。たとえば初対面でマイノリティの特性をカミングアウトしたり、n 人で行ったミーティングの議事録や結果を勝手に公開したり、個別チャットではなく皆に見えるチャンネル内でやりとりしたがったりする。

エクスポーザーはメンバーに以下の不安を抱かせる:

- 自分の情報も晒されるんじゃないか……
- この人はこんなに開示してくれてるので、自分ももっと開示しないといけないんじゃないか……
- この人の水準が「当たり前」に昇格するんじゃないか。情報発信というコストのかかる作業が今後タスクとして整備されてしまうんじゃないか……

対処としては、まず情報公開と自己開示の塩梅を言語化することである。エクスポーザーの良し悪しはチームの状況次第であり、エクスポーザー自体が悪いわけではない。チームの状況にそぐわず、晒しすぎに寄るから悪いのである。

したがって、チームの状況をまずは言語化し、その上で、何が良くて何がダメかまで明らかにする。そうしない限り、エクスポーザーが収まることはない。エクスポーザー自体は悪いことではないからだ。むしろ現代の仕事観として優れているまである。「DM だと負えないし政治的になるからチャンネルに書いて」などは、Slack を使うチームなら同意できるだろう。しかし、だからといって、センシティブなカミングアウトをいきなりチャンネルでやられても（チーム内の動揺や当惑が大きく）困るわけである。バランスが大事であり、バランスを知るために現状の言語化が必要なのである。この手間を惜しんではならない。

## アンリーダブル（わかりづらすぎ）
チームメンバーとの直接的なコミュニケーションが難しい者を指す。

チームにおけるコミュニケーションはメッシュ的――どのメンバーが、どのメンバーに対しても直接的にコミュニケーションできることが前提である。しかしアンリーダブルはそれを行えるだけの能力や要領がなく、翻訳役が介在することとなる。新人のオンボーディングなど立ち上がりの段階では望ましいが、そうではなく、立ち上がった後の話だ。この構図の問題は単純で、翻訳役の負担が高すぎることにある。厄介なことに、翻訳役はチーム内で最も優秀なエースが担当することも多く、つまりはエースを食いつぶしてしまう。あるいは、楽観的に見ても、エースが翻訳を軽視または放棄して、アンリーダブルが空き状態となる。仕事をしていない空きメンバーは、チームの緊張を嫌な方向に緩めることも多い――要は手抜きが加速する。

アンリーダブルの対処はハードスキル、ソフトスキル、ドメイン知識の三軸で考えると良い。

### ハードスキル
ハードスキルが足りなくて「何がわからないのかがわからない」状態に陥ることはよくある。この場合は、ハードスキルそのものを鍛えさせるしかない。Slack や GitHub や Claude Code を当たり前に使う組織において、それらを自律的に使えないのなら、その時点で論外であり、使えるようになるまで鍛えるしかない。必要なら業務時間を鍛錬時間にしてでも鍛えてもらうべきだ。無論、ハードスキルは選考段階で判別できるのが理想である。

難しいのはチーム側のハードスキルが遅れている場合である。GitHub と Markdown に熟知したエンジニアが、Excel と共有フォルダを使っているチームに入ったとしよう。この場合、新人には Excel と共有フォルダのスキルを鍛えてもらうべきと言いたいところだが、新人からすると「前時代的なやり方自体が悪い」と見えて、モチベーションが湧かない。このような状況を軽減する策は現状ない。こうなった時点で詰んでいるので、こうならないよう選考を行うべきだ。つまり選考時は、ハードスキルとして何を使っているかを伝えねばならない。

### ソフトスキル
次にソフトスキルについては、口頭と読み書きに二分できる。60 分の議論を口頭だけで行う場合と、読み書きだけで行う場合を考えてほしい。口頭スキルまたは読み書きスキルがあるとは、それらができる水準を指す。おそらく大半のチームはどちらか片方に偏る。経験上、読み書きスキルを持つチームは稀少である。

そういうわけで、まずはアンリーダブルは口頭スキルが弱いと考えて良い。

アンリーダブルの対処としても、当然ながら口頭スキルを身に付けてもらうことになる。まずはメンバーとコミュニケーションを行う場数を踏ませる。ファシリテーションは、慣れたメンバーが行うのが良いだろう。それで特性的に出来るかどうかを判断する。

できそうにない場合は、残念ながら口頭スキルだけではまかなえないので、読み書きスキルを使う方向に寄せねばならない。たとえば従来だと 1 日 3 時間くらい口頭で会話するチームがあったとしよう。アンリーダブルな新人は、特性的にそこまではこなせず、1 日 1 時間くらいが限度かもしれない。単純計算で、2 時間くらいコミュニケーション密度が足らないが、これは読み書きで補う必要がある。あるいは、コミュニケーションの量を少し免除しなければならない。

### ドメイン知識
ドメイン知識とはそのチームが抱えている固有の文化、背景、その他知識全般を指す。


まわりがある。口頭コミュニケーションメインのチームだと、読み書きしかできない者はアンリーダブルになりがちだ。逆に読み書きメインのチームだと、口頭しか使えない者がアンリーダブルになる。こちらについてもハードスキルと同様、


は、場数を増やせて鍛えることである。各メンバーとの直接的なコミュニケーションの機会を

積ませるのである。積ませて、それでも成長できない場合は、チームのオンボーディングに問題がある。

そのためにはアンリーダブルが主体的にタスクに従事できるよう、オンボーディングを完了していなければならない。そうでなければ OJT 的につきっきりでフォローするしかなくなってしまい、これはフォローする側が忙しいため容易に破綻する。
